The Cure, Part 2: Debugging with Al
When you encounter a bug, your first instinct might be to paste the code into an Al and say, “Fix this.”
This is a mistake. It reinforces code blindness and often leads to superficial patches rather than true
solutions. Instead, use Al as your “Rubber Duck”—a partner to explain the problem to you.
The Process:
1. Isolate: Identify the exact error message and stack trace.
2. Explain: Prompt the Al to explain why the error is happening in the context of your code.
3. Analyze: Review the Al's explanation. Does it make sense? Is it pointing to a fundamental logic
error?
4. Implement: Apply the fix once you understand the root cause.

The Cure, Part 3: Refactoring for Maintainability
Refactoring is the process of restructuring existing code without changing its external behavior. It's how
we fight “code rot” and make our app maintainable for the long term.
The Golden Rule: DRY (Don't Repeat Yourself)
Al loves to repeat itself. It will happily generate the same 50-line card layout in three different files
instead of creating one reusable component. Your job is to spot these patterns and eliminate duplication.
A Key principle: If your Ul file contains business logic or API calls, refactor immediately. This is a
violation of “Separation of Concerns.”
